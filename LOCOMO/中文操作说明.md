## 1. 现在数据是什么状态

我已经把原来的知识库、聊天记录、会话日志清掉了。

现在系统里只导入了 LOCOMO：

```text
10 个 LOCOMO 对话样本
5882 条对话记忆块
1986 条 QA 问题
```

切分方式是：

```text
LOCOMO 官方的一句话 / 一个 dia_id = 一个 ARPM 聊天记忆块
```

也就是说，不再按知识库父子块切分。

这样做是为了方便评测，因为 LOCOMO 官方答案证据就是 `dia_id`。

## 1.1 向量模型与隔离索引

当前已经把三套新增向量模型拉到本地，和原中文模型放在同一个父目录：

```text
E:\flai\ARPM-v4备份\assets\models
```

模型清单在：

```text
E:\flai\ARPM-v4备份\assets\models\embedding_models.json
```

已建好 LOCOMO 隔离向量库的模型是：

```text
bge-large-en-v1.5
bge-m3
all-MiniLM-L6-v2
```

隔离索引位置是：

```text

```

每套模型各自一份 FAISS，不和原来的中文模型、不和其他模型混用。

前端控制台的“向量模型”下拉框可以切换模型。中文 LOCOMO 优先建议先用 `bge-m3`，因为它是多语种模型。

如果你以后重新导入 LOCOMO 或改了切分方式，需要重建三套隔离索引：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\build_model_indexes.py --models bge-large-en-v1.5 bge-m3 all-MiniLM-L6-v2 --batch-size 16
```

## 2. 你第一步应该跑什么

先跑“只检索，不调用 AI”的测试。

这个测试不花 API 钱，主要看系统能不能把官方证据找回来。

命令：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_retrieval_eval.py --method chat_vector --k 20
```

跑完后，会生成两个结果：

```text
LOCOMO\results\retrieval_chat_vector.jsonl
LOCOMO\results\retrieval_chat_vector.summary.json
```

其中：

```text
retrieval_chat_vector.jsonl
```

是一行一题的详细结果。

```text
retrieval_chat_vector.summary.json
```

是总分。

## 3. 生成你能看的中文表格

检索跑完后，执行：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\make_zh_report.py --input LOCOMO\results\retrieval_chat_vector.jsonl
```

会生成：

```text
LOCOMO\results\retrieval_chat_vector.zh.csv
```

这个文件可以用 Excel 打开。

里面有这些列：

```text
QA编号
样本
类别
英文问题
官方答案
官方证据dia_id
召回dia_id
Hit@1
Hit@5
Hit@10
MRR
Top5召回片段
```

你主要看：

```text
Hit@5
Hit@10
Top5召回片段
```

如果 `Hit@5` 是“是”，说明前 5 个召回里找到了官方证据。

如果 `Hit@10` 是“是”，说明前 10 个召回里找到了官方证据。

如果都是“否”，说明这题没把正确证据召回。

## 5. 三种检索方法是什么意思

当前脚本支持三种检索方式。

### 5.1 chat_vector

命令：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_retrieval_eval.py --method chat_vector --k 20
```

意思：

```text
只用聊天历史向量检索。
```

这是最基础的 ARPM 聊天记忆检索基线。

建议第一个跑它。

### 5.2 arpm_retrieval

命令：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_retrieval_eval.py --method arpm_retrieval --k 20
```

意思：

```text
走 ARPM retriever 接口，但只启用聊天历史，不启用知识库。
```

这个比 `chat_vector` 更接近系统真实流程。

### 5.3 arpm_temporal

命令：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_retrieval_eval.py --method arpm_temporal --k 20
```

意思：

```text
走 ARPM retriever 接口，并启用时间权重。
```

这个用于看“时间权重”是否提升 LOCOMO 证据召回。

## 6. 跑 AI 生成答案之前，先看检索

不要一上来就跑 AI 生成答案。

原因：

```text
如果检索没有召回正确证据，AI 很可能答不对。
```

推荐顺序：

```text
第一步：跑 chat_vector 检索
第二步：看中文 CSV
第三步：再跑 arpm_retrieval
第四步：再跑 arpm_temporal
第五步：挑 20 条跑 AI 生成答案
第六步：确认没问题再跑全量 QA
```


## 8. 三种生成方法是什么意思

### 8.1 plain_rag

命令：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_qa_generation.py --method plain_rag --limit 20
```

意思：

```text
普通 RAG 问答。
只给模型 LOCOMO 任务说明 + 召回片段 + 问题。
```

这是普通 RAG 基线。

### 8.2 arpm_protocol

命令：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_qa_generation.py --method arpm_protocol --limit 20
```

意思：

```text
加入 ARPM 分析式协议。
```

结构是：

```text
ARPM 分析式协议
+ LOCOMO 任务说明
+ 召回片段
+ 问题
```

### 8.3 arpm_full

命令：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_qa_generation.py --method arpm_full --limit 20
```

意思：

```text
完整 ARPM 方式。
```

结构是：

```text
ARPM 分析式协议
+ ARPM 记忆注入规则
+ LOCOMO 任务说明
+ 按时间顺序排列的召回记忆
+ 问题
```

这里会保证：

```text
轮次早的召回内容在上面
轮次晚的召回内容在下面
每块都有 physical_time
最后读到的是召回集合里最近期的内容
```

## 9. 如果你要使用自己原来的 prompt

把你的原始 prompt 保存成一个 txt 文件，例如：

```text

```

然后运行：

```powershell
E:\flai\ARPM-v4备份\env\arpm-venv\Scripts\python.exe LOCOMO\run_qa_generation.py --method arpm_full --limit 20 --system-prompt-file LOCOMO\my_system_prompt.txt
```

这样你的原始 prompt 会放在最前面。

后面再接：

```text
ARPM 分析式协议
ARPM 记忆注入规则
LOCOMO QA 任务说明
召回内容
问题
```

## 10. 怎么看 summary.json

例如：

```text
LOCOMO\results\retrieval_chat_vector.summary.json
```

里面主要看：

```text
recall_at_5
recall_at_10
recall_at_20
mrr
```

含义：

```text
recall_at_5：前 5 个召回里是否包含官方证据
recall_at_10：前 10 个召回里是否包含官方证据
recall_at_20：前 20 个召回里是否包含官方证据
mrr：正确证据排得越靠前越高
```

如果是 QA 生成结果，还会有：

```text
em
f1
```

含义：

```text
em：答案是否完全匹配
f1：答案词级别相似度
```

## 11. 一个重要提醒

当前使用的向量模型是：

```text
shibing624/text2vec-base-chinese
```

但 LOCOMO 是英文数据。

所以第一轮检索分数可能偏低。

这不是脚本坏了。

这反而可以作为一个实验点：

```text
中文向量模型检索英文 LOCOMO 的弱基线
```

后面可以换英文或多语向量模型，再做对照实验。
