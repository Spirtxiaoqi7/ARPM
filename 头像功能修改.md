# 头像功能修改

## 1. 需求目标

本次只为聊天风格页面增加“用户头像 / AI 角色头像自定义上传”功能。头像属于当前会话的 UI 展示配置，不是用户身份字段，不是 AI 角色配置字段，也不参与聊天生成、RAG、历史记忆或向量索引。

## 2. 修改范围

- `backend/api/session.py`
- `backend/web/templates/index.html`
- `backend/web/static/js/chat.js`
- `backend/web/static/css/style.css`
- `start.bat`
- `start.sh`
- `头像功能修改.md`

## 3. 后端修改

在 `backend/api/session.py` 中新增会话头像相关能力：

- 新增资源归属上下文函数 `get_current_owner_context(request)`，当前返回 `owner_scope=local`、`owner_id=default`。
- 新增统一路径函数：
  - `get_avatar_storage_dir(owner_context, session_id)`
  - `get_avatar_public_url(owner_context, session_id, filename)`
  - `build_avatar_storage_key(owner_context, session_id, filename)`
  - `assert_avatar_session_access(owner_context, session_id)`
  - `verify_avatar_modify_permission(request, owner_context, session_id)`
- 新增上传接口 `POST /api/session/<session_id>/avatar`。
- 新增删除 / 恢复默认接口 `DELETE /api/session/<session_id>/avatar?role=user|assistant`。
- 新增静态访问路由 `GET /uploads/avatars/<path:storage_key>`。
- `/api/history/<session_id>` 返回中增加 `ui_settings`，用于前端切换会话时读取头像配置。

头像保存路径：

```text
runtime/arpm-app/data/avatar_uploads/<owner_scope>/<owner_id>/sessions/<session_id>/
```

当前未登录系统时实际路径：

```text
runtime/arpm-app/data/avatar_uploads/local/default/sessions/<session_id>/
```

上传安全限制：

- `ENABLE_AVATAR_UPLOAD` 必须为 `true`，否则上传和删除接口拒绝。
- 允许格式：`png`、`jpg`、`jpeg`、`webp`。
- 禁止 SVG 和其他非图片文件。
- 最大文件大小：2MB。
- 文件名由后端生成，不使用上传原始文件名。
- 静态访问只允许读取 `avatar_uploads` 根目录下的图片文件。

## 4. 前端修改

在 `backend/web/templates/index.html` 的常规设置弹窗中：

- “用户设定”区域下方新增“用户头像”设置项。
- “AI角色设定”区域下方新增“AI 角色头像”设置项。
- 每个设置项包含预览、上传按钮、恢复默认按钮和隐藏文件选择框。

在 `backend/web/static/js/chat.js` 中：

- 新增 `State.avatars = { user, assistant }`。
- 新增头像上传、删除、预览和消息刷新函数。
- 切换会话时从 `data.ui_settings.avatars` 读取当前会话头像。
- 新建会话时头像回到默认状态。
- 消息渲染时按当前会话头像显示用户消息和 AI 回复。
- 头像上传使用独立接口，不复用 `/api/history`，不写入 messages。

在 `backend/web/static/css/style.css` 中：

- 新增头像设置项样式。
- 新增头像预览样式。
- 新增聊天消息头像图片样式。

本地启动脚本调整：

- `start.bat` 增加 `if not defined ENABLE_AVATAR_UPLOAD set "ENABLE_AVATAR_UPLOAD=true"`。
- `start.sh` 增加 `export ENABLE_AVATAR_UPLOAD="${ENABLE_AVATAR_UPLOAD:-true}"`。
- 这只影响通过本地脚本启动的开发环境；外部预先设置的环境变量仍然优先生效。
- `docker-compose.yml` 未改，容器部署仍需显式配置后才开启上传。

## 5. 数据字段

session JSON 新增字段：

```json
{
  "ui_settings": {
    "avatars": {
      "user": {
        "url": "/uploads/avatars/local/default/sessions/<session_id>/user_avatar_xxx.png",
        "storage_key": "local/default/sessions/<session_id>/user_avatar_xxx.png",
        "filename": "user_avatar_xxx.png",
        "role": "user",
        "updated_at": "2026-05-16Txx:xx:xxZ"
      },
      "assistant": {
        "url": "/uploads/avatars/local/default/sessions/<session_id>/assistant_avatar_xxx.png",
        "storage_key": "local/default/sessions/<session_id>/assistant_avatar_xxx.png",
        "filename": "assistant_avatar_xxx.png",
        "role": "assistant",
        "updated_at": "2026-05-16Txx:xx:xxZ"
      }
    }
  }
}
```

旧 session 没有 `ui_settings` 时，前端按空头像配置处理，继续显示默认“我 / AI”。

## 6. 未来登录系统预留设计

- 当前未实现登录系统。
- 当前 `owner_scope = local`。
- 当前 `owner_id = default`。
- 前端不传 `owner_id`。
- 头像归属完全由后端 `get_current_owner_context(request)` 决定。
- 头像路径通过 owner context 统一生成，没有在多个位置散落拼接。
- 未来登录系统接入后，可将 `get_current_owner_context(request)` 替换为 `owner_scope=user`、`owner_id=current_user.id`。
- 当前 `assert_avatar_session_access(owner_context, session_id)` 已预留 session 访问校验入口。
- 当前 `verify_avatar_modify_permission(request, owner_context, session_id)` 已预留头像修改权限入口。
- 当前未新增 users 表，未新增注册页，未新增登录页，未写死用户名、邮箱、手机号或昵称。

## 8. 上线部署影响

- 后端代码层面 `ENABLE_AVATAR_UPLOAD` 默认关闭。
- 本地 `start.bat` / `start.sh` 已默认设置 `ENABLE_AVATAR_UPLOAD=true`，便于本地开发和演示上传头像。
- 生产环境如需允许上传，需要显式设置 `ENABLE_AVATAR_UPLOAD=true`，尤其是 Docker / 服务管理器 / Nginx 后端进程环境。
- 读取和展示已有头像不受该开关影响。
- 当前 `docker-compose.yml` 已挂载 `./runtime:/app/runtime`，因此 `runtime/arpm-app/data/avatar_uploads` 会随 runtime 持久化。
- 如果生产环境前面有 Nginx / OpenResty，需确保 `/uploads/avatars/` 转发到 Flask 应用，或按同等安全规则只暴露 avatar upload 根目录下的图片文件。
- 文件大小限制为 2MB。
- 文件格式限制为 `png/jpg/jpeg/webp`。
- 未配置 `ENABLE_AVATAR_UPLOAD=true` 时，普通公网访问者无法通过头像接口上传或删除头像。

## 10. 风险说明

- 文件上传能力默认关闭，部署时必须明确设置环境变量后才允许修改头像。
- 当前只做轻量图片魔数校验和扩展名校验，不做图片重编码。
- 删除 / 恢复默认只清空 session JSON 中对应头像配置，不强制删除物理图片，避免误删风险；长期运行后可增加安全清理任务。
- 旧会话没有 `ui_settings` 时按默认头像处理。
- 未来接入登录系统时，应扩展 owner context 和 session ownership 校验，不应让前端传 owner id。

## 11. 主题头像隔离修改

- 聊天风格和学术风格仍然是同一套业务逻辑，同一个 `session_id` 下不复制会话、不复制 `messages`、不复制历史记忆、不复制 RAG 召回链路、不复制向量索引。
- 头像按 `ui_settings.theme_profiles.chat` 和 `ui_settings.theme_profiles.research` 隔离保存。
- 新上传头像写入 `ui_settings.theme_profiles.<theme>.avatars.<role>`，其中 `theme` 只允许 `chat` 或 `research`，`role` 只允许 `user` 或 `assistant`。
- 旧数据中的 `ui_settings.avatars` 兼容读取为 chat 主题头像；research 主题不会自动复用旧头像或 chat 头像。
- 后端保存路径按主题隔离：
  - `runtime/arpm-app/data/avatar_uploads/local/default/sessions/<session_id>/themes/chat/`
  - `runtime/arpm-app/data/avatar_uploads/local/default/sessions/<session_id>/themes/research/`
- 切换主题只切换 UI 头像来源，聊天内容、RAG、历史记忆、生成逻辑、协议诊断和向量索引都不受影响。

## 12. 会话显示标签与日志安全

- `session_name` 是现有会话主名称，仍由“新建会话命名 / 重命名会话”接口维护。
- `session_label` 是新增独立备注字段，只用于 UI 展示和日志辅助说明。
- `display_title` 是前端展示标题。
- `display_title` 去重规则：
  - `session_label` 为空时只显示 `session_name`
  - `session_label == session_name` 时只显示 `session_name`
  - `session_label != session_name` 时显示 `session_name（session_label）`
- 新增接口 `PATCH /api/session/<session_id>/label` 只更新 `session_data["session_label"]`，允许清空，不修改 `session_name`、`user_name`、`character_name`、`messages`、历史记忆或向量索引。
- 日志文件名不使用 `session_label`。
- 日志内容可以记录 `session_label` 作为安全辅助字段。
- `session_label` 不参与 prompt、messages、RAG、向量索引、导出主文件名或日志文件路径。

## 13. 新版日志字段复用说明

- 日志系统增强已存在，本次不重复实现 `admin_logger`。
- 本次复用 `schema_version`、`event_id`、`trace_id`、`owner_scope`、`owner_id`、`session_name`、`session_label`、`display_title`、`theme`。
- A/B/C 三类业务审计日志保持文件名不变：
  - `A_recall.log`
  - `B_dialog.log`
  - `C_cot.log`
- JSONL 格式保持不变。
- 原字段保持不变，不删除、不重命名、不改变含义。
- `avatar_url`、`storage_key`、本地头像路径不写入 A/B/C 正文。
- 头像上传、头像删除、标签修改等运行状态只写入 `app.log` 摘要，摘要只包含 `session_id`、`theme`、`role`、成功/失败状态和错误原因概要。
