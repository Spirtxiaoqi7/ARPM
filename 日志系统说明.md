# ARPM-v4 日志系统说明

## 目录职责

`runtime/arpm-app/admin/` 是业务审计日志目录，用于保存对话业务过程中的关键审计记录，支持业务复盘、实验追踪、Bad Case 分析和 prompt 注入内容检查。

`runtime/arpm-app/logs/` 是系统运行日志目录，用于保存应用运行状态日志，例如服务启动、模块加载、索引构建、检索数量、向量索引保存和重要错误排查信息。

## 日志文件

`A_recall.log` 记录召回内容日志，主要保存本轮实际注入 prompt 的召回上下文。它可能包含知识库召回和历史对话召回。新增的 `recall_summary` 只作为摘要辅助筛选，不替代原有 `context` 字段。

`B_dialog.log` 记录对话输入输出日志，主要保存用户输入、AI 回复、模型、轮次、角色名称和生成状态摘要。

`C_cot.log` 记录分析和协议诊断日志，主要保存 `analysis`、`regeneration_info`、`protocol_info` 以及用于筛选的协议和重生摘要字段。

`app.log` 是应用运行日志，路径为 `runtime/arpm-app/logs/app.log`。它记录服务运行、模块加载、索引构建、错误排查等运行状态。它不是对话审计日志，不应写入完整用户对话、完整召回上下文或完整协议分析内容。

`app.log` 当前格式为 `time | level | module | message`，使用 UTF-8 追加写入。当前已低风险接入 `backend/app.py`、`backend/api/chat.py`、`backend/core/retriever.py`、`backend/storage/vector_store.py`、`backend/storage/memory_store.py` 等关键位置，用于记录服务启动、日志模块版本、模型加载、知识库索引加载、BM25 构建、RAG 召回数量、向量索引保存和重要错误信息。

服务启动时会写入一条诊断日志，例如 `admin logger loaded schema_version=2.0 module_path=...`，用于确认当前运行中的后端进程已经加载增强后的 `admin_logger.py`。如果业务审计日志仍是旧格式，可以先检查 `app.log` 中是否存在这条启动诊断记录，并确认服务是否已重启。

## JSONL 格式

JSONL 的中文名是逐行 JSON 日志格式。每一行都是一条完整 JSON 记录，文件整体不是一个 JSON 数组。这种格式适合追加写入，也适合按行读取、筛选和导入分析工具。

本项目的 `A_recall.log`、`B_dialog.log`、`C_cot.log` 均继续使用 JSONL，追加写入，不重写历史旧日志。

## 新增字段含义

`schema_version`，中文名：日志结构版本号。表示当前日志记录使用的字段结构版本，用于未来日志字段升级时区分新旧格式。当前示例值为 `"2.0"`。

`event_id`，中文名：单条日志事件 ID。每一条日志记录的唯一 ID，用于排查单条日志，不同日志行不能重复。当前使用 UUID。

`trace_id`，中文名：链路追踪 ID / 单轮追踪 ID。用于把同一轮回答中的召回日志、对话日志、协议诊断日志串联起来。当前格式为 `<session_id>::round_<round>::<model>`。如果缺少值，会回退为 `unknown_session`、`round_unknown`、`unknown_model`。

`owner_scope`，中文名：资源归属范围。为未来登录系统预留。当前没有登录系统时固定为 `"local"`，未来多用户系统中可以改为 `user`、`team`、`org` 等。

`owner_id`，中文名：资源归属 ID。为未来登录系统预留。当前没有登录系统时固定为 `"default"`，未来登录系统中可以改为真实 `user_id`。

`session_name`，中文名：会话主名称。稳定的主会话名称，例如“对话1”“对话2”，用于稳定识别会话。它不应被 `session_label` 替代。

`session_label`，中文名：会话显示标签 / 会话备注名。用户额外设置的备注名，例如“航天基地”“产品面试准备”。它只用于前端展示和日志辅助说明，不参与 prompt、RAG、messages、向量索引或日志文件名。

当前前端的“新建会话命名 / 重命名会话”保存到后端 `session_name`。因此新增日志会保留 `session_id` 作为稳定 ID，同时把前端自定义会话名写入 `session_name`；在尚未实现独立备注字段 `session_label` 时，`session_label` 会回退为同一个前端自定义会话名，方便日志查看和检索。未来如果增加独立备注字段，则优先使用真实 `session_label`。

`display_title`，中文名：前端展示标题。由 `session_name` 和 `session_label` 拼接出来的人类可读标题。例如 `session_name = "对话1"`、`session_label = "航天基地"` 时，`display_title = "对话1（航天基地）"`。

`theme`，中文名：当前视觉主题。记录当前前端主题，只允许使用 `chat` 或 `research`。`chat` 表示聊天风格，`research` 表示学术风格。`theme` 只是 UI 上下文，不得影响 RAG、prompt、生成逻辑或评估指标。

`input_length`，中文名：用户输入长度。记录 `B_dialog.log` 中用户输入文本长度，便于筛选异常长输入。

`reply_length`，中文名：AI 回复长度。记录 `B_dialog.log` 中 AI 回复文本长度，便于筛选空回复或异常长回复。

`generation_success`，中文名：生成是否成功。记录本轮生成是否成功返回可用结果。

`is_regenerate`，中文名：是否重新生成。`dialog_turn` 为 `false`，`dialog_regenerate` 为 `true`。

`protocol_status`，中文名：协议状态摘要。由 `protocol_info` 摘要生成，用于快速筛选是否需要协议修复。

`repair_status`，中文名：协议修复状态摘要。由 `protocol_info` 摘要生成，用于快速查看是否已修复或仍需修复。

`analysis_visible`，中文名：是否存在分析标签。由 `protocol_info` 摘要生成，用于筛选模型是否按协议输出分析标签。

`regeneration_used`，中文名：是否使用重新生成。由 `regeneration_info` 摘要生成，用于筛选本轮是否发生过重生尝试。

`recall_summary`，中文名：召回摘要。记录知识库召回数量、历史对话召回数量、是否注入上下文、BM25/RRF 配置、top_k 和相似度阈值等摘要信息。无法获取的值可以为空，不硬编码假数据。

## 原有字段含义

`logged_at`，中文名：日志写入时间。记录该条日志实际写入文件的时间。

`event`，中文名：事件类型。表示这条日志是什么事件，例如 `recall_context_injected`、`dialog_turn`、`dialog_regenerate`、`cot_analysis`、`cot_regenerate`。

`session_id`，中文名：会话 ID。后端用于识别会话的稳定 ID，不能用 `session_label` 替代。

`round` / `current_round`，中文名：对话轮次。表示当前是第几轮对话。如果原代码中 `A_recall.log` 使用 `current_round`，`B_dialog.log` 和 `C_cot.log` 使用 `round`，应继续保留原字段。

`physical_time`，中文名：物理时间。表示对话发生时的业务时间或消息时间。

`model`，中文名：模型名称。表示本轮调用的大模型名称，例如 `deepseek-chat`。

`user_name`，中文名：用户名称。参与角色对话和生成逻辑的用户名称，不得被 `session_label` 替代。

`character_name`，中文名：AI 角色名称。参与角色对话和生成逻辑的 AI 角色名称，不得被 `session_label` 替代。

`context`，中文名：实际注入 Prompt 的召回上下文。`A_recall.log` 中记录的召回内容，包含知识库召回和历史对话召回。

`analysis`，中文名：分析内容 / 协议分析内容。`C_cot.log` 中记录的 analysis 信息。

`regeneration_info`，中文名：重新生成信息。记录本轮是否启用重新生成、尝试次数、最大尝试次数、是否成功等。

`protocol_info`，中文名：协议诊断信息。记录 `analysis_tag`、`response_tag`、`visible_reply`、`needs_repair`、`was_repaired` 等协议状态。

## 兼容性原则

本次增强只追加字段，不删除、不重命名、不改变原字段含义。这样旧分析脚本仍然可以继续读取 `logged_at`、`event`、`session_id`、`round`、`current_round`、`physical_time`、`model`、`user_name`、`character_name`、`user_input`、`assistant_reply`、`context`、`analysis`、`regeneration_info`、`protocol_info` 等字段。

`A_recall.log`、`B_dialog.log`、`C_cot.log` 文件名保持不变，是为了避免破坏已有运维脚本、实验分析脚本和历史数据路径约定。`session_label`、`display_title`、头像和 `theme` 都不得参与业务审计日志文件名生成。

历史旧日志不会被重写。历史乱码可能来自旧数据写入方式或控制台编码链路，本次修改只保证新增日志使用 UTF-8 写入，不对旧日志做破坏性修复。

## owner_context 扩展

当前系统没有登录系统，`owner_scope` 和 `owner_id` 由后端统一决定，固定为：

```json
{
  "owner_scope": "local",
  "owner_id": "default"
}
```

未来接入登录系统时，可以只替换后端的 owner context 获取函数，例如：

```json
{
  "owner_scope": "user",
  "owner_id": "current_user.id"
}
```

当前不新增登录系统、不新增用户表、不新增注册页、不新增登录页，也不从前端接收 `owner_id`。
